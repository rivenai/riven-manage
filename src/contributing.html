Riven AI/Contributing

Contributing to Riven AI

Riven AI is open source and welcomes contributions of all kinds — bug fixes, new providers, documentation improvements, plugins, and more. This guide covers everything you need to start contributing.

maximhq/rivenDiscord Community

Ways to Contribute

Bug Reports

Found a bug? Open a GitHub issue with a minimal reproduction case.

New Providers

Add support for an LLM provider not yet in Riven AI. Providers follow a standard interface.

Plugins

Build pre/post-LLM plugins to extend Riven AI's request pipeline.

Performance

Profile and optimize hot paths. Riven AI is latency-sensitive; every nanosecond matters.

Documentation

Improve existing docs, add examples, or write tutorials for common use cases.

SDKs

Improve or create client SDKs in languages not yet covered.

Development Setup

Prerequisites: Go 1.22+, Docker, Make, and golangci-lint.

setup
# Clone the repository
git clone https://github.com/maximhq/riven.git
cd riven-api

# Download Go dependencies
go mod download

# Start development dependencies (Redis, etc.)
docker-compose -f docker-compose.dev.yml up -d

# Run tests to verify setup
go test ./...

# Build the binary
go build -o riven-api ./cmd/riven-api

# Run with the example config
./riven-api --config config.example.json

Useful Makefile targets:

bash
make dev          # Run with hot reload (requires 'air')
make lint         # Run golangci-lint
make test         # Run unit tests
make test-integration  # Run integration tests (requires running providers)
make test-coverage     # Run tests with HTML coverage report
make build        # Build release binary

Branch Strategy

BranchPurpose
mainStable release branch. PRs targeting main must include tests and docs.
developIntegration branch for upcoming releases. Start feature branches from here.
feature/*New features. Branch from develop, PR back to develop.
fix/*Bug fixes. Branch from main for hotfixes, develop for regular fixes.
docs/*Documentation-only changes. Can PR directly to main.

Adding a Provider

Every provider implements the Provider interface in providers/. The interface requires three methods:

providers/my_provider.go
package providers

type MyProvider struct {
    baseURL string
    client  *http.Client
}

// GetProviderKey returns the provider identifier used in config
func (p *MyProvider) GetProviderKey() string {
    return "my-provider"
}

// TransformRequest converts a Riven AI-internal request to provider format
func (p *MyProvider) TransformRequest(req *schema.BifrostRequest) ([]byte, error) {
    // Map BifrostRequest fields to your provider's API format
    payload := MyProviderRequest{
        Model:    req.Model,
        Messages: transformMessages(req.Params.ChatCompletionParams.Messages),
    }
    return json.Marshal(payload)
}

// TransformResponse converts provider response to Riven AI-internal format
func (p *MyProvider) TransformResponse(body []byte, model string) (*schema.BifrostResponse, error) {
    var raw MyProviderResponse
    if err := json.Unmarshal(body, &raw); err != nil {
        return nil, err
    }
    return &schema.BifrostResponse{
        Choices: transformChoices(raw.Choices),
        Usage:   transformUsage(raw.Usage),
    }, nil
}
After implementing the interface, register your provider in providers/registry.go and add a corresponding entry to the model catalog in config/models.go. Add at least one integration test in providers/my_provider_test.go.

Pull Request Process

  1. Fork the repository and create your branch from develop (or main for hotfixes)
  2. Write tests for new functionality — aim for > 80% coverage on new code
  3. Run make lint test and ensure all checks pass
  4. Update docs in docs/ if your change affects the public API or user-visible behavior
  5. Commit using Conventional Commits format: feat(providers): add Mistral support
  6. Open the PR with the provided template filled out — describe the problem, solution, and any trade-offs
  7. Address review feedback; squash commits before final merge if requested

Code Style

  • Run gofmt and goimports before committing — the CI will reject unformatted code
  • Keep functions under ~60 lines; prefer many small, focused functions
  • Avoid global state; pass dependencies explicitly via constructor injection
  • Document all exported types, functions, and constants with GoDoc comments
  • Prefer table-driven tests using testify for assertions
  • Use context propagation for cancellation and deadlines on all I/O operations

Performance Contributions

Riven AI is benchmarked at 5,000+ req/s with sub-millisecond overhead. Performance PRs must include benchmark results:

bash
# Run benchmarks
go test -bench=. -benchmem -run=^$ ./...

# Compare before and after with benchstat
go install golang.org/x/perf/cmd/benchstat@latest
go test -bench=. -count=5 -run=^$ ./... > before.txt
# make your change
go test -bench=. -count=5 -run=^$ ./... > after.txt
benchstat before.txt after.txt

Include the benchstat output in your PR description. Regressions > 2% will block merging without justification.
